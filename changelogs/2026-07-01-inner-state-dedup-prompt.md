# 心声防重复 — 提示词注入最近 5 条 innerState

**日期**：2026-07-01  
**涉及 commit**：`08c888e`

## 改了什么

心声（innerState）由主 LLM 在每次回复末尾附 `<emotion>...</emotion>` 块生成。**LLM 完全看不到自己之前写过什么**，所以连续 3 轮内心独白可能都是"她在忙""还没回""等她回"——内容 90% 相似。

修复：在 `[心声输出要求]` 段后追加 `[最近心声]` 段，把 `char.emotionHistory` 最近 5 条 innerState 注入 system prompt（带触发时间，如"3分钟前"），并明确要求：

- 不重复同样的事
- 不重复同样的情绪
- 不继续上一条没解决的悬念

## 动了哪些文件

- `hooks/useChatAI.ts` —— `[心声输出要求]` 段后追加 `[最近心声]` 段（line 803 之后，14 行新增）

## 踩坑 / 需要知道的（重要）

### LLM 不会自觉避免重复，除非你告诉它之前说过什么

之前的提示词只说"必须附加 `<emotion>` 块"，但没把历史心声传给 LLM。这意味着 LLM 每次都是"白纸状态"写 innerState——它不知道 3 轮前写过类似的。

修复后，system prompt 里会看到类似：

```
[最近心声 — 本次 innerState 必须跟以下明显不同...]
1. （3分钟前）她又没回……表情包算回复吗？
2. （15分钟前）手机屏幕暗下去又亮起来，每次以为是她……
```

LLM 有了上下文，自然会写不同角度的内容。

### 为什么是最近 5 条而不是 1-2 条

暮色试过 1 条、2 条都太短——LLM 只知道"上一条"，可能会写"上一条的镜像"（比如上一条是"她在忙"，这次写"她可能忙完了"——仍然是同一个事）。

5 条能覆盖更长的情绪/话题上下文，LLM 能识别出"我一直在想她回不回消息"这个模式，主动换一个维度（比如想自己的事、想今天要做的事、想之前聊过的话题）。

**token 成本**：5 条 innerState × 平均 50 字 ≈ 250 字中文，prompt 里多 250 字对大部分 LLM 可忽略。

### 时间戳为什么用相对分钟数

绝对时间戳（如 `2026-07-01 18:35:12`）对 LLM 没用——它不知道当前时间。相对分钟数（"3 分钟前""15 分钟前"）让 LLM 知道"这是最近的、连着的"。

**注意**：这个相对时间是客户端计算的，LLM 看到的"3 分钟前"是我这边 Date.now() - createdAt 算出来的。如果 LLM 用绝对时间反推，可能跟客户端时间有偏差——但这不影响判断，只影响 LLM 能不能精确推断"什么时候发生的事"。

### 没加代码层去重（暮色选了纯方案 A）

讨论时我提了 3 个方案：
- A. 提示词注入（依赖 LLM 自觉）
- B. 代码层相似度检测（不依赖 LLM）
- C. A+B 组合（双保险）

暮色选了纯 A——理由应该是改动小、出问题好回滚。如果 model 还是偷懒重复，再加 B 兜底也不晚。

**判断阈值**：如果试几次发现某些 model 还是容易重复（比如小模型 / 量化模型），再加 `jaccardBigramSimilarity` 函数做兜底。预期阈值 0.7——高于 0.7 视为"换汤不换药"。

### 不要顺便优化心声数量限制

`emotionHistory.slice(0, 100)` 是历史保留 100 条上限。这次没动它——暮色只关心"不要重复"，没说"历史保留少一点"。如果以后觉得 100 条太多（IndexedDB 占用 / 弹窗加载慢），再单独评估。

## 备注

- 这个改动**只在主 LLM 的 system prompt 里加了约 250 字**，不影响：
  - 副 API（情绪评估、记忆宫殿、彼方等）
  - buffs 列表（心声卡片那套）——这些是另一套机制，跟 innerState 写入独立
  - 日程、聊天历史、消息列表
- 触发条件：`isEmotionOn(char)` 为 true 才注入——心声关闭时不浪费 token
- 没写测试（项目本身没有测试体系），靠暮色真实使用验证效果
